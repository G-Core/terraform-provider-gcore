Handling Default “Hidden” Security Group Rules in Terraform Providers

Background: Default Rules in Cloud Security Groups

Many cloud platforms automatically include default firewall rules when a new security group (or similar resource) is created. For example, AWS VPC security groups get a default “allow all outbound” rule, and Azure NSGs come with built-in allow/deny rules. These “hidden” rules are not explicitly defined by the user but exist on the backend. This poses a challenge for Terraform provider design: how to represent or handle these default rules without forcing users to declare them in code.

Problems with hidden default rules: If the provider does nothing special, Terraform will detect these backend-added rules as unmanaged changes (drift). Users would see Terraform trying to remove or change rules they never wrote in their config. Conversely, requiring users to manually include the provider’s default rules in every config is tedious and defeats the purpose of having sensible defaults.

Best Practices from Existing Providers

Terraform providers for major platforms have converged on clear patterns for handling default security group rules:

AWS – Removing Default Egress Rule

AWS by default adds an “ALLOW ALL” egress rule to new VPC security groups. The Terraform AWS provider’s solution is to automatically remove that default rule on creation, and require users to explicitly add any outbound rule they want. As the docs note:

“By default, AWS creates an ALLOW ALL egress rule when creating a new Security Group in a VPC. When Terraform creates the Security Group, it will remove this default rule, requiring you to explicitly recreate it if desired. We feel this leads to fewer surprises in controlling your egress rules.” ￼

In other words, when you use aws_security_group in Terraform, it starts with no rules at all – you must declare even the “allow all outbound” if you want it. This design choice improves clarity and security. AWS’s console chose convenience (auto-allow outbound), but Terraform chose explicitness:

“Terraform…has made the decision the other way…suits the tool better as well as slightly improving security posture (at the expense of making people define a repeated egress block).” ￼

Deletion vs. keep toggle: The AWS provider does this unconditionally (there is no toggle to keep the default rule). If a user does want an allow-all egress, they just add an egress rule for 0.0.0.0/0 in their config. This way, the Terraform state matches exactly what’s in the config. (AWS even provides a separate aws_default_security_group resource for managing the VPC’s default SG, which on adoption immediately deletes all default rules so Terraform can manage them explicitly ￼.)

OpenStack – Following the AWS Pattern

OpenStack Neutron security groups similarly auto-create an allow-all egress rule. Initially, this caused issues: Terraform would create a group with user-specified rules, and Neutron would add a wide-open egress rule that overrode the intended restrictions. A user report highlighted that:

“On security group creation, a default egress ANY:ANY is added by Neutron next to the one managed by TF. This makes it impossible to manage egress rules from TF, since they are overridden by the default egress rule.” ￼

The resolution was to mirror AWS’s approach – the OpenStack provider was updated to automatically delete the default egress rule on creation. The issue reference notes: “This was fixed for the AWS provider… Is it possible to have the same fix for OpenStack?” ￼. By removing the implicit rule, Terraform ensures only the user-declared rules remain.

DigitalOcean – No Implicit Allow (Default is Deny)

DigitalOcean’s cloud firewall behaves a bit differently: if you don’t specify an outbound rule, no outbound traffic is allowed (default deny). In practice, their web UI adds a permissive outbound rule by default for convenience, but the underlying model is “deny by default.” Terraform’s DigitalOcean provider requires you to declare whatever outbound rules you need. The documentation states clearly:

“Outbound firewall rules define traffic allowed to leave. If no outbound rules are configured, no outbound traffic is permitted.” ￼

Thus, there isn’t a hidden rule that Terraform must handle – you either add an allow-all rule yourself or your droplet can’t send out traffic. This is effectively an explicit approach by design, so Terraform just mirrors that (no special provider logic needed aside from possibly guiding users to include an outbound rule if desired).

Azure – Built-in NSG Rules (Can’t be Removed, Only Overridden)

Azure Network Security Groups come with several default rules (e.g. allow VNet traffic, allow Azure load balancer, deny all inbound, allow all outbound, etc.). These cannot be deleted – they always exist at lower priority. Terraform’s Azure provider doesn’t expose these as part of the azurerm_network_security_group resource’s rule blocks at all. They’re simply documented and always present. As a result, users don’t declare them in Terraform; you only define your custom rules, which can override defaults by using higher priority values. Microsoft’s guidance:

“Azure creates default rules in each NSG… You cannot remove the default rules, but you can override them by creating rules with higher priorities.” ￼

Terraform just leaves those default NSG rules alone under the hood. There’s no drift because the provider doesn’t try to manage or track them – it knows they’re “built-in”. The user isn’t forced to deal with them in code. (If a user wants to negate a default allow rule, they must add their own deny rule with a higher priority, since the defaults stay in effect otherwise.)

Cloudflare – Managed Default Rulesets

While Cloudflare doesn’t have “security groups,” it has WAF Managed Rulesets that are enabled by default. By default Cloudflare enables standard rule sets (like OWASP core rules). The Terraform Cloudflare provider handles these via resources like cloudflare_ruleset and cloudflare_waf_rule to override or disable specific managed rules. The key is that the default rules exist implicitly; if users want to modify them, they do so by declaring override resources rather than listing every default rule. The provider knows the defaults and doesn’t require all of them to be in the config. For example, to disable a specific managed WAF rule, you can use a Terraform resource to set an override, while the rest of the managed rules remain managed by Cloudflare internally.

Design Considerations for Provider Developers

1. Don’t force users to declare backend-imposed defaults. The consensus best practice is that a Terraform config should only need to include the resources and rules the user cares about. Default rules that the cloud always adds should either be automatically removed or ignored by the provider. This avoids a situation where the user must copy-paste “magic” rules just to appease the provider.

2. If default rules are removable – remove them by default. As seen with AWS and OpenStack, if the cloud API allows those rules to be deleted, the provider should do so (ideally right after resource creation). This gives the user a clean slate. It also effectively means the user can “delete” or change those rules via Terraform: by simply not defining them (Terraform will ensure they don’t exist), or by defining a replacement. For example, AWS’s approach means the user can omit any egress rules to have a locked-down SG, or add one if needed – either way Terraform has full control ￼. This approach leads to least surprise: the rules in the cloud = the rules in your .tf file.

3. If default rules cannot be removed – treat them as immutable and out-of-band. In cases like Azure NSGs, the provider should mark these as inherent defaults and not attempt to manage them. Document their existence, but do not include them in state or plan output. This way, Terraform plans don’t constantly show changes. Users handle them only by overrides (e.g. adding a higher-priority rule to counter a default allow). Terraform, in effect, ignores the default entries. This is usually implemented by filtering out any default rules in the read logic of the resource.

4. Provide toggles or separate resources if needed. In rare cases, a provider might allow toggling the behavior. For instance, a community Terraform module for an OpenStack-based cloud includes a delete_default_rules = true flag to remove the allow-all rules ￼ ￼. In the official providers, such toggles are uncommon – they choose a sane default and stick to it. If you suspect some users may want to retain the default rules, you could expose a parameter (e.g. keep_default_rules = true), but note that this complicates the state handling significantly. The simpler pattern is always remove if removable, since users can always add back an equivalent rule explicitly if desired.

5. Clearly document the behavior. Whatever approach you take, document it in the provider docs. For example, the AWS provider’s docs have a prominent note about the default egress removal ￼. This heads off confusion when users notice the cloud console vs Terraform behavior differ. Documentation should explain if Terraform will auto-delete any rules, or if some rules are implicitly present but not managed.

Trade-offs and User Experience

The main trade-off in handling hidden rules is between user convenience vs explicit control:
	•	Cloud consoles often add permissive defaults to avoid breaking things (e.g. allowing outbound so new VMs can download updates). Terraform prioritizes a predictable, declarative state. It opts for explicitness even if it means a bit more initial work for the user (for example, adding an egress rule in every SG). The benefit is that users won’t be surprised by unseen rules allowing traffic ￼, and security posture can be stricter by default.
	•	By not forcing users to declare the hidden defaults, we keep configs cleaner. Imagine if AWS required every Terraform SG resource to include an egress block just to silence a diff – that would be noisy. Instead, Terraform just removes the AWS default so no diff occurs. The user only writes what they want, and Terraform makes reality match it.
	•	If a user wants to manage or remove a default rule, the provider’s strategy should enable that. In AWS/OpenStack, deletion is automatic (thus Terraform-managed by default). In Azure, since deletion isn’t possible, the next best thing is allowing an override rule. In Cloudflare, you can use resources to disable specific managed rules. The goal is that any rule affecting the cloud resource is either represented in Terraform or deliberately ignored, but never left in a half-managed limbo.
	•	Drift prevention: Providers often implement custom logic in the CRUD operations to handle these rules. For AWS, right after creating the SG via API, the provider calls RevokeSecurityGroupEgress for 0.0.0.0/0 if it sees it, so that Terraform state starts clean. This prevents a situation where a terraform plan right after apply shows a change to remove a default rule. The AWS provider’s code ensures the default rules don’t enter the state at all (or are immediately removed) ￼. Similar logic can be applied in other providers.

In summary, the best practice is to manage “hidden” default rules in Terraform behind the scenes so that the user’s config remains a source of truth for what rules actually exist. For G-Core’s security group resource specifically, you should likely implement the AWS-style approach: on creation, check for any default rules the platform auto-adds (perhaps a default allow outbound or intra-group rule) and remove them (via the API) if possible. Document this behavior. This way, the Terraform resource will only reflect user-defined rules. If the backend’s default rules are meant to be permanent and not removable, then treat them like Azure: don’t require them in config, and perhaps automatically mark them as computed in state so Terraform doesn’t plan their removal. But generally, if G-Core allows deletion of those default rules, it’s cleaner to have Terraform do so, rather than forcing the user to manage a “phantom” rule they never wrote.
