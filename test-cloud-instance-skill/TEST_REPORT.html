Test Description

This test imports an existing instance and checks for drift. A successful import should show "No changes" on subsequent plan.

Drift Issues Found

Field Issue Impact
project_id Set to 379987 in state after import, but null in config (uses env var) forces replacement
region_id Set to 76 in state after import, but null in config (uses env var) forces replacement
volumes.boot_index Not read from API during import, shows as added drift

Terraform Plan Output After Import

# gcore_cloud_instance.test_external must be replaced
-/+ resource "gcore_cloud_instance" "test_external" {
      - project_id = 379987 -> null  # forces replacement
      - region_id  = 76 -> null      # forces replacement
      ~ volumes = [
          ~ {
              + boot_index = 0       # drift - not read from API
            },
        ]
    }

Root Cause Analysis

1. project_id/region_id issue: These fields are marked as "forces replacement" when null in config but present in state. The config relies on environment variables (GCORE_CLOUD_PROJECT_ID, GCORE_CLOUD_REGION_ID) but after import, explicit values are in state.

2. boot_index issue: The instance GET API doesn't return boot_index for volumes. The import reads volumes but can't populate boot_index, causing drift.

Recommended Fixes

  1. Make project_id and region_id not force replacement when transitioning from explicit to environment-variable-based
  2. During import, fetch volume details separately to get boot_index, OR document that users must specify boot_index in config before import
