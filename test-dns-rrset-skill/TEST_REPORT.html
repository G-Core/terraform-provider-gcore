Issues Found

Meta drift caused by apijson.Unmarshal CRITICAL

Description: Read method used apijson.Unmarshal which populates ALL fields from API response, including optional-tagged meta fields. DNS server injects extra meta keys via FillMetaDefaults(), so the API response meta differs from what user specified.

Root Cause: apijson.Unmarshal overwrites user-set meta with API response containing server-injected defaults.

Fix: Changed to apijson.UnmarshalComputed which only populates computed and computed_optional fields, preserving user's original meta.

Status: FIXED & VERIFIED

UseStateForUnknown on user-managed computed_optional fields WARNING

Description: Fields warning, warnings, limit, strict, and filter_set_id had UseStateForUnknown plan modifier. This causes: (1) stale values persisting across updates, (2) attribute deletion not being detected.

Impact: Stale warnings would persist after the underlying issue is resolved. Removing pickers limit/strict would not send the change to API.

Fix: Removed UseStateForUnknown from all affected fields. Only truly stable computed fields (name, type) retain it.

Status: FIXED & VERIFIED

Known Issues / Expected Behavior

  • Record ID changes on update: DNS API assigns new record IDs on every PUT/Replace operation. This is expected — id field in resource_records is computed and shows (known after apply) during updates.
  • DNS API uses PUT not PATCH: The Rrsets.Replace SDK method sends a full PUT request. MarshalJSONForUpdate handles computing the diff between plan and state.
  • Pickers strict defaults to false: When creating a picker without specifying strict, the API returns strict: false. This is correctly stored as a computed value.
  • DELETE operations not in MITM: The destroy test DELETE calls were not captured in the MITM flow file (proxy may have been restarted). Destroy was confirmed successful by Terraform output.
