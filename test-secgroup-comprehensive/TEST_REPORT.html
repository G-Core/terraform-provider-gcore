Issues Found

Issue 1: 409 Conflict on Parallel Rule Operations Medium

When Terraform creates or destroys multiple rules targeting the same security group in parallel, the v2 async endpoints return 409 Conflict because the SG resource is locked by an in-flight task.

POST /cloud/v2/security_groups/{project}/{region}/{sg_id}/rules
409 Conflict: "Resource `{sg_id}` is already locked by another task."
  • Impact: Partial failure on first apply when creating/deleting many rules
  • Workaround: Run terraform apply again, or use -parallelism=1
  • JIRA: Comment added to GCLOUD2-20783, related to GCLOUD2-5064

Default Rules Auto-Deletion

The Gcore API auto-creates ~40 default egress rules when a security group is created. Our implementation:

  • Creates SG via NewAndPoll() (waits for async task)
  • Fetches the SG to discover default rules
  • Deletes each rule via Rules.DeleteAndPoll()
  • Sets security_group_rules to empty list in state

Timing: ~2m28s for create + deletion of 40 async delete operations

Config Validator

The rulesNotAllowedValidator blocks the inline rules attribute at validation time, directing users to gcore_cloud_security_group_rule:

Error: Inline rules are not supported

Use the gcore_cloud_security_group_rule resource to manage security group
rules instead of inline rules. This ensures each rule has its own lifecycle
and prevents conflicts between the parent security group and individual
rule resources.

RequiresReplace Strategy (GCLOUD2-22560)

All mutable fields on gcore_cloud_security_group_rule have RequiresReplace(), meaning:

  • Changing any rule attribute triggers destroy + create (new rule ID)
  • The buggy PUT endpoint (500 error) is never called
  • Matches the API's actual behavior: PUT performs a replace operation
