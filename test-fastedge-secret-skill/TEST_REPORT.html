Issues Found

Import requires one apply to stabilize Set identity WARNING

Description: After importing a fastedge_secret by ID, the first terraform plan shows a change in secret_slots. One apply is needed to sync the Set internal representation.

Root Cause: SetNestedAttribute uses internal element identity for comparison. The imported state's Set representation differs from the one built by ModifyPlan. This is a known Terraform framework behavior with Set attributes.

Impact: Minor - import works correctly but requires one apply after import to stabilize. No data loss or incorrect behavior.

Recommendation: Document in resource docs that terraform apply should be run once after import.

ModifyPlan makes extra GET call per plan INFO

Description: ModifyPlan fetches live API checksums on every plan/apply. This adds one extra GET request per operation but is necessary to work around SetNestedAttribute's semantic equality swapping element mappings.

Impact: Minimal - one extra lightweight GET call per plan operation.

Known Issues / Expected Behavior

  • API GET response lacks id field: The FastEdge Secrets API does not return the secret ID in GET/PATCH responses. The provider preserves and restores the ID from state. Handled correctly.
  • API PATCH response returns empty secret_slots: When secret_slots are not in the PATCH body, the response returns "secret_slots":[]. The provider correctly ignores this and preserves state slots.
  • secret_slots is a SetNestedAttribute: Set semantics mean element order is not guaranteed. The provider handles this via ModifyPlan identity preservation.
