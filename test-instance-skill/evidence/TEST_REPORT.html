Bugs Found (3)

🐛 BUG #1: Flavor Change Task Not Awaited

File: internal/services/cloud_instance/resource.go

Description: When changing flavor (e.g., g1-standard-1-2 → g1-standard-2-4), the provider returns before the changeflavor task completes. This causes "inconsistent result" errors because the API still reports the old flavor when state is refreshed.

Expected: Provider should use ChangeFlavorAndPoll to wait for task completion.

Actual: Provider reads state immediately, gets old flavor value.

Error: Provider produced inconsistent result after apply

When applying changes to gcore_cloud_instance.test, provider
"registry.terraform.io/gcore/gcore" produced an unexpected new value:
.flavor: was cty.StringVal("g1-standard-2-4"), but now
cty.StringVal("g1-standard-1-2").

🐛 BUG #2: Tags Model Type Can't Handle Unknown Values

File: internal/services/cloud_instance/model.go:45

Description: The Tags field uses *map[string]types.String which cannot represent unknown values in Terraform's type system.

Expected: Tags field should use types.Map or customfield.Map to handle unknown values.

Actual: When tags variable is not set or computed, Terraform fails with:

Error: Value Conversion Error

Received unknown value, however the target type cannot handle unknown values.
Path: tags
Target Type: *map[string]basetypes.StringValue
Suggested Type: basetypes.MapValue

Suggested Fix: Change Tags *map[string]types.String to Tags types.Map in model.go.

🐛 BUG #3: boot_index Not Populated During Import

File: internal/services/cloud_instance/resource.go (ImportState)

Description: After importing an instance, the boot_index field in volumes is not read from the API. This causes drift on subsequent plans.

Expected: After import, terraform plan should show no changes.

Actual: Terraform plan shows drift:

  ~ volumes = [
      ~ {
          + boot_index = 0
          # (1 unchanged attribute hidden)
        },
    ]

Impact: Users importing existing instances will always see drift on first plan.

Deep Dive: Tags Bug Analysis

Root Cause Analysis

The Mismatch:

Component Current Implementation Required Implementation
Schema (schema.go:209) schema.MapAttribute with ElementType: types.StringType Correct
Model (model.go:32) *map[string]types.String customfield.Map[types.String] or types.Map

Why This Fails

In Terraform Framework, when a variable is not yet known (e.g., default = null or computed from another resource), it's represented as an "unknown" value. The type *map[string]types.String is a Go native map and cannot represent this unknown state.

// CURRENT (buggy) - Go native map can't be "unknown"
Tags *map[string]types.String `tfsdk:"tags" json:"tags,optional,no_refresh"`

// FIXED - customfield.Map has UnknownMap() support
Tags customfield.Map[types.String] `tfsdk:"tags" json:"tags,optional,no_refresh"`

Scope of Impact

This affects ALL resources using tags with the same pattern:

  • cloud_instance
  • cloud_load_balancer
  • cloud_volume
  • cloud_network
  • cloud_security_group
  • cloud_floating_ip
  • cloud_file_share
  • cloud_network_subnet
  • cloud_gpu_virtual_cluster
  • And more...

How Old Provider Handled It

The old provider used Terraform SDK v2:

// Old provider (SDK v2) - schema.TypeMap with Go map[string]string
"metadata_map": &schema.Schema{
    Type:     schema.TypeMap,
    Optional: true,
    Elem: &schema.Schema{
        Type: schema.TypeString,
    },
}

SDK v2's schema.TypeMap handles unknown values internally. The new Terraform Framework requires explicit handling via types.Map or customfield.Map.

Fix Location

This is a Stainless code generator issue. When generating model types for MapAttribute fields, it should use customfield.Map[T] instead of *map[string]T.

The customfield package already exists at /internal/customfield/map.go with proper support:

// From customfield/map.go - already handles unknown values!
func UnknownMap[T attr.Value](ctx context.Context) Map[T] {
    return Map[T]{MapValue: basetypes.NewMapUnknown(elemType[T](ctx))}
}
