Fixes Applied During Testing

Fix 1: WriteOnly not allowed in SetNestedAttribute

Issue: Terraform framework does not allow WriteOnly: true inside SetNestedAttribute. Schema validation fails at provider startup.

Root Cause: Set semantics require hashing all child attributes for uniqueness, but WriteOnly attributes can't be hashed (never stored in state).

Fix: Changed secret_slots from SetNestedAttribute to ListNestedAttribute and changed value from WriteOnly: true to Sensitive: true. Also updated model type from NestedObjectSet to NestedObjectList.

Files:

  • internal/services/fastedge_secret/schema.go:33 - SetNestedAttribute → ListNestedAttribute
  • internal/services/fastedge_secret/schema.go:51 - WriteOnly:true → Sensitive:true
  • internal/services/fastedge_secret/model.go:15 - NestedObjectSet → NestedObjectList

Fix 2: ID lost during Read/Update (API doesn't return ID)

Issue: The FastEdge GET and PATCH responses do not include id field. The apijson.UnmarshalComputed decoder sets computed fields not present in JSON to null, causing id to be lost.

Root Cause: GET /fastedge/v1/secrets/{id} returns {"app_count":0,"name":"...","secret_slots":[]} without id. PATCH response also omits id.

Fix: Preserve ID before API call and restore after unmarshal in Read, Update, and ImportState.

Files:

  • internal/services/fastedge_secret/resource.go:152 - Read: save/restore ID
  • internal/services/fastedge_secret/resource.go:100 - Update: save/restore ID
  • internal/services/fastedge_secret/resource.go:252 - ImportState: restore ID

Fix 3: ImportState used UnmarshalComputed instead of Unmarshal

Issue: ImportState was using UnmarshalComputed which skips required and optional fields (like name and comment), resulting in incomplete state after import.

Fix: Changed ImportState to use apijson.Unmarshal (full deserialization) since all fields need to be populated on import.

Files:

  • internal/services/fastedge_secret/resource.go:246 - Unmarshal instead of UnmarshalComputed

Fix 4: Read used Unmarshal instead of UnmarshalComputed

Issue: Read function was using apijson.Unmarshal which would overwrite user-provided values (like name) with API values. Should preserve user config values.

Fix: Changed Read to use apijson.UnmarshalComputed which only updates computed fields from API response.

Files:

  • internal/services/fastedge_secret/resource.go:169 - UnmarshalComputed for Read

Known Issues / Warnings

Warning 1: Create API response is incomplete

The POST response only returns {"name":"...","id":N} without app_count and secret_slots. This causes a one-time output drift on the first plan after create. A second apply resolves it. This is an API-side issue, not a provider bug.

Warning 2: WriteOnly + Set incompatibility required List fallback

The ideal design would use SetNestedAttribute (no ordering) with WriteOnly for secret values. However, Terraform framework explicitly forbids WriteOnly children in Set attributes. The fallback is ListNestedAttribute + Sensitive, which means: (1) ordering matters - changing slot order causes drift, (2) secret values are stored in state (masked as sensitive).

Warning 3: GET/PATCH responses don't include ID

The FastEdge API GET and PATCH responses omit the id field. Only the POST (create) response includes it. This required manual ID preservation in Read, Update, and Import functions. This should be reported as an API improvement suggestion.
